This guide walks you through deploying Trenova locally using Docker and getting to a running system in a single script. The setup takes around five to ten minutes depending on your internet connection and hardware.

## Prerequisites

Before you begin, confirm your environment meets the minimum requirements.

**Hardware:**

| Resource | Minimum | Recommended |
|----------|---------|-------------|
| CPU | 4 cores | 8 cores |
| RAM | 8 GB | 16 GB |
| Storage | 50 GB SSD | 100 GB SSD |

**Software:**

| Tool | Version |
|------|---------|
| Docker Engine | 24.0+ |
| Docker Compose | v2 (plugin or standalone) |
| OpenSSL | Any |

Trenova runs on Linux (Ubuntu 22.04+, Debian 12+, RHEL 8+) and Windows Server 2019+ with WSL2.

## Deploy Trenova

<Steps>
  <Step title="Clone the repository">
    Clone the Trenova repository and navigate to the `deploy` directory:

    ```bash
    git clone https://github.com/emoss08/trenova-2.git
    cd trenova-2/deploy
    ```
  </Step>

  <Step title="Run the setup script">
    Execute the setup script. It handles everything from credential generation to database seeding:

    ```bash
    ./setup.sh
    ```

    The script will:

    1. Check that Docker, Docker Compose, and OpenSSL are installed.
    2. Prompt you for a domain name (press **Enter** to use `localhost`).
    3. Generate cryptographically secure credentials for all services.
    4. Build and start all Docker containers.
    5. Run database migrations against PostgreSQL.
    6. Seed the database with initial reference data.

    <Tip>
      If you use a custom domain instead of `localhost`, add both the main domain and the `storage.` subdomain to your DNS or hosts file before running the script — for example, `127.0.0.1 trenova.local` and `127.0.0.1 storage.trenova.local`.
    </Tip>
  </Step>

  <Step title="Access the application">
    Once the script finishes, open your browser and go to `http://localhost`. Log in with the default admin credentials:

    | Field | Value |
    |-------|-------|
    | Email | `admin@trenova.app` |
    | Password | `admin123!` |

    <Note>
      Change the default admin password immediately after your first login. The default credentials are well-known and should never be used in any environment accessible over a network.
    </Note>

    The following service URLs are available on a localhost deployment:

    | Service | URL |
    |---------|-----|
    | Application | `http://localhost` |
    | API health | `http://localhost/api/v1/health` |
    | Temporal UI | `http://localhost/temporal-ui` |
    | MinIO console | `http://localhost/minio` |
  </Step>

  <Step title="Change your password">
    In the application, navigate to your user profile and update the password before doing anything else. This replaces `admin123!` with a credential only you know.
  </Step>

  <Step title="Explore the application">
    After changing your password, you can start exploring the platform:

    - **Shipments** — Create a new shipment by navigating to Dispatch → Shipments. You will need at least one customer and one service type configured first.
    - **Fleet** — Add tractors, trailers, and equipment types under the Fleet section.
    - **Workers** — Onboard drivers and assign them to moves under the Workers section.

    <Tip>
      The API health endpoint at `http://localhost/api/v1/health` confirms the backend is running correctly. Check it if any service appears unavailable.
    </Tip>
  </Step>
</Steps>

## Production deployment

For production, use pre-built images from GitHub Container Registry instead of building locally:

```bash
cd deploy

# Copy and configure the environment file
cp .env.example .env
# Edit .env with your domain and review the generated secrets

# Set the version and pull images
export TRENOVA_VERSION=latest
docker compose -f docker-compose.prod.yml pull
docker compose -f docker-compose.prod.yml up -d
```

Production images are published at `ghcr.io/emoss08/trenova-2/tms` and tagged with semantic versions (e.g., `1.0.0`) as well as `latest`.

<Note>
  For full production setup including TLS configuration, reverse proxy settings, backup procedures, and update management, see the [Deployment overview](/deployment/overview).
</Note>
