Trenova uses a resource-based access control model. Every user belongs to an organization and carries one or more role assignments. Roles are named collections of resource permissions that determine what a user can see and do.

## User accounts

Navigate to **Admin â†’ Users** to create and manage user accounts.

### User fields

| Field | Description |
|---|---|
| **Name** | Full display name (max 255 characters) |
| **Username** | Unique login handle, 1â€“20 characters |
| **Email Address** | Used for login and notifications |
| **Timezone** | The user's local timezone, used for displaying timestamps |
| **Time Format** | `12-hour` or `24-hour` clock preference |
| **Status** | `Active` or `Inactive` â€” inactive users cannot log in |
| **Profile Picture** | Optional avatar image |
| **Must Change Password** | Forces a password reset on next login |
| **Is Locked** | Locks the account, preventing login regardless of status |

### Account status rules

- A user with **Status = Inactive** receives an authorization error on login.
- A user with **Is Locked = true** is blocked even if their status is `Active`. Use the lock flag when you need to temporarily suspend access without deactivating the account.
- Only platform administrators (users with `isPlatformAdmin = true`) can unlock accounts.

### Organization membership

A user can belong to multiple organizations. Each membership record tracks:
- Which **organization** the user is a member of
- Which **business unit** the user belongs to within that organization
- When the user **joined** and who **granted** the membership
- An optional **expiry date** after which the membership is automatically revoked

The user's **current organization** is the one they are actively working in. Users switch organizations from their profile menu.

## Roles

A role is a named collection of resource permissions. Roles are scoped to an organization.

Navigate to **Admin â†’ Roles** to create and manage roles.

### Role fields

| Field | Description |
|---|---|
| **Name** | Display name for the role (max 255 characters) |
| **Description** | Optional free-text explanation of the role's purpose |
| **Is System** | System roles are created by Trenova and cannot be deleted |
| **Is Org Admin** | Grants organization-wide administrative access |
| **Is Business Unit Admin** | Grants administrative access scoped to a business unit |
| **Max Sensitivity** | The highest field sensitivity level this role can access: `public`, `internal`, `restricted`, or `confidential` |
| **Parent Role IDs** | Optional list of roles whose permissions this role inherits |

<Note>
  System roles (`isSystem = true`) are managed by Trenova. You can assign them to users but cannot modify or delete them.
</Note>

### Role hierarchy

Roles support inheritance via **Parent Role IDs**. A role automatically inherits all permissions from its parent roles. You can build a hierarchy where a senior dispatcher role inherits all permissions of a junior dispatcher role and adds additional ones.

### Operations

Each resource permission grants a set of operations. Operations have dependency rules â€” granting a higher-privilege operation automatically includes its dependencies.

| Operation | Dependencies | Description |
|---|---|---|
| `read` | â€” | View records |
| `create` | `read` | Create new records |
| `update` | `read` | Edit existing records |
| `delete` | `read` | Delete records |
| `export` | `read` | Export records to file |
| `import` | `read`, `create` | Bulk import records |
| `approve` | `read`, `update` | Approve pending records |
| `reject` | `read`, `update` | Reject pending records |
| `assign` | `read`, `update` | Assign resources (e.g., assign driver to shipment) |
| `unassign` | `read`, `update` | Remove an assignment |
| `archive` | `read`, `update` | Archive a record |
| `restore` | `read`, `update` | Restore an archived record |
| `submit` | `read`, `update` | Submit for review or approval |
| `cancel` | `read`, `update` | Cancel a record |
| `duplicate` | `read`, `create` | Duplicate a record |
| `close` | `read`, `update` | Close a record |
| `lock` | `read`, `update` | Lock a record against further edits |
| `unlock` | `read`, `update` | Unlock a locked record |
| `activate` | `read`, `update` | Activate an inactive record |
| `reopen` | `read`, `update` | Reopen a closed or cancelled record |

<Note>
  When you grant an operation, Trenova automatically grants all of its dependencies. When you revoke an operation, Trenova automatically revokes all operations that depend on it.
</Note>

### Data scope

Each resource permission also has a **data scope** that controls which records the role can access:

| Scope | Description |
|---|---|
| `own` | Only records created by or assigned to the user |
| `organization` | All records within the user's current organization |
| `all` | Records across all organizations (platform admin only) |

### Field sensitivity

Roles have a **Max Sensitivity** level that controls which sensitive fields are visible:

| Level | Description |
|---|---|
| `public` | Freely visible data |
| `internal` | Internal operational data |
| `restricted` | Data restricted to authorized roles |
| `confidential` | Highly sensitive data such as payroll and personal information |

A role with `maxSensitivity = internal` cannot see fields marked as `restricted` or `confidential`, even if the role has `read` permission on that resource.

## Assigning roles to users

To assign a role to a user:

1. Navigate to **Admin â†’ Users** and open the user's profile.
2. Select **Add Role** and choose the role from the list.
3. Optionally set an **Expiry Date** â€” after this date the assignment is automatically revoked.

To remove an assignment, open the user's profile, find the role in the **Roles** tab, and select **Remove**.

You can also manage assignments from **Admin â†’ Roles** by selecting a role and navigating to its **Assignments** tab.

## Admin vs. regular users

There are two special flags that grant elevated access outside the normal permissions model:

- **Is Platform Admin** (`isPlatformAdmin`) â€” Grants unrestricted access to all organizations and all resources on the platform. Assign this only to Trenova system administrators.
- **Is Org Admin** (on a role with `isOrgAdmin = true`) â€” Grants administrative access to all resources within the user's current organization.

<Warning>
  Platform admin access bypasses all resource permission checks. Grant it only to trusted infrastructure personnel.
</Warning>

## Session management

Trenova tracks the last login time for each user (`lastLoginAt`). Administrators can view this from the user's profile to identify inactive accounts.

Session timeout is configured at the platform level. Users are automatically signed out after the configured idle period.
